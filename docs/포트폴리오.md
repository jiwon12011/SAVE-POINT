# SAVE POINT — 포트폴리오 케이스 스터디

> **지친 하루를 RPG 상태창으로 번역하고, 오늘 가능한 "최소 퀘스트"를 제안하며, NPC 조력자가 회복 대사를 건네는 서브컬처 루틴 앱.**
>
> *"오늘의 나를 저장하고, 다시 시작할 힘을 얻는 곳."*

---

## 0. 30초 요약 (TL;DR)

루틴·감정기록 앱은 **잘 해내는 날**을 위해 설계된다. 그래서 정작 사람들이 앱을 닫는 **망한 날**엔 죄책감의 원천이 된다. SAVE POINT는 그 반대로 설계했다 — **망한 날에 오히려 켜고 싶은 앱.**

그걸 가능하게 한 **3가지 핵심 설계 결정:**

| 결정 | 무엇 | 왜 |
|---|---|---|
| **규칙 엔진 + LLM 하이브리드** | 규칙이 난이도·구조를 결정론적으로 정하고, LLM은 *문구만* 채운다 | 일관성·오프라인 동작·예측가능성 + AI의 자연스러움을 동시에 |
| **Zero-failure streak** | "세이브=완료 아님" — 퀘스트 0개여도 streak 유지 | 이탈 없는 습관. "오늘도 실패"가 아니라 "오늘도 저장" |
| **데이터 주도·제공자 독립 설계** | 대사·퀘스트·보상·상태이상을 전부 JSON 데이터로 분리 | AI 없이도(P0) 전 루프 동작, LLM 제공자 교체 자유 |

> **한 줄 정체성:** *상태를 게임처럼 가볍게 바라보게 만들어, 컨디션이 나빠도 "내가 할 수 있는 만큼만 해도 괜찮다"는 경험을 주는 앱.*

## 2. 문제 정의 (Why)

### 2-1. 기존 앱의 공통된 실패
투두 앱은 못 한 일을 쌓아 보여준다. 무드 트래커는 번아웃이 왔을 때 이모지 하나 고를 기력조차 요구한다. 루틴 앱은 하루 빠지면 빨간 X를 찍는다.

**공통점: 전부 "잘 해내는 날"을 위해 설계됐다.** 컨디션 좋은 날엔 도파민을 주지만, 사람들이 앱을 닫는 날은 컨디션이 나쁜 날이다. 결국 사용자가 앱을 그만 쓰는 이유는 하나로 수렴한다 — **"잘 못 하면 쓰기 싫어지는 앱이라서."**

### 2-2. 타깃 사용자 (페르소나)
- **지수 (25, 디자인 전공 졸업유예생)** — 포폴 마감 3주 전. 매일 밤 투두를 쓰지만 아침엔 아무것도 하기 싫다. 루틴 앱 스트릭은 이미 끊겼고, 공들인 Notion 대시보드는 열면 죄책감만 든다. *페인포인트: 못 하고 있다는 증거를 자꾸 기록하게 되는 앱.*
- **민재 (29, 3년차 개발자)** — 번아웃인지 아닌지도 모르는 상태. 퇴근 후 소파에서 폰만 보다 잔다. 헬스·독서·영어를 동시에 시작했다 2주 만에 다 포기. 게임 세계관엔 몰입하는데 현실 루틴엔 그런 몰입이 없다. *페인포인트: 의지력을 전제하는 앱 구조.*

### 2-3. 경쟁 앱 대비 차별점
| 앱 | 그들의 한계 | SAVE POINT |
|---|---|---|
| **Finch** | 새가 자라려면 결국 목표 달성 필요 (못 한 날은 제자리) | 아무것도 못 한 날의 체크인도 유효한 세이브 |
| **Habitica** | 할 일 안 하면 HP 깎이는 패널티 + 높은 진입장벽 | 패널티 없는 RPG. 못 깨도 상태창은 정직하게 남음 |
| **무드 트래커(Daylio 등)** | 감정 수집에서 끝, "그래서 뭘?"에 답 없음 | 상태 → 퀘스트 → 대사로 이어지는 행동 흐름 |
| **일반 투두/루틴** | 할 일 중심 → 못 하는 날 죄책감의 원천 | 상태 중심 → 컨디션 먼저 진단, 그에 맞는 최소 행동 |

### 2-4. 핵심 차별화 4가지
1. **"아무것도 못 해도 세이브는 된다"** — 완료가 아니라 체크인 자체가 세이브. 이탈 없는 습관.
2. **금지 퀘스트** — "오늘은 이걸 하지 마라"는 부정 미션(야식 금지, 무한 스크롤 금지). 번아웃 회복엔 *해야 할 것보다 하지 말 것*이 더 실용적일 때가 많다.
3. **상태이상 태그 — 감정을 진단명으로** — "우울하다" 대신 "수면 부족 / 과부하 Lv.2". 자책 언어를 기술 언어로 치환하면 감정적 거리두기가 된다.
4. **NPC 조력자 다양성** — 같은 상황에 힐러는 따뜻하게, 라이벌은 도발적으로. 사람마다 회복에 필요한 언어가 다르다.

## 3. 핵심 사용자 경험 (User Flow)

### 3-1. 전체 루프
```
앱 열기 → [체크인] → [상태창 생성] → [퀘스트] → [NPC 대화] → [세이브] → [기록 누적]
           HP/MP/집중력     규칙엔진이        모드별        상태 맞는      완료 0개여도
           +상태이상+보스    난이도 결정      퀘스트        NPC 회복대사   세이브 가능
```

### 3-2. 첫 실행 (1회)
스플래시 → 로그인(또는 **게스트 시작**) → 온보딩(이름·직업 별명·자주 걸리는 상태이상·선호 NPC) → 첫 체크인 → 첫 상태창 → 첫 퀘스트 → 첫 세이브. `onboarded` 플래그로 두 번 다시 안 나옴.

### 3-3. 매일 반복 루프 + 분기
앱 진입 시 위에서 먼저 걸리는 분기를 실행:
```
세션 없음 → 로그인 / 온보딩 안 됨 → 온보딩
오늘 체크인 안 함 → 홈(체크인 유도)   ← 보통 여기
오늘 체크인 함   → 홈(오늘 상태창 재열람)
```
홈 상단엔 **연속 세이브 배너**.

| 단계 | 사용자 | 시스템 |
|---|---|---|
| **체크인**(30초) | HP/MP/집중력 4단계 + 상태이상 다중선택 + **오늘의 보스(선택)** | 입력 저장 |
| **상태창** | RPG 상태창 확인 | 규칙 엔진이 에너지 점수 → 난이도 모드 계산 |
| **퀘스트** | 메인/서브/금지 확인·체크 | 모드 슬롯에 문구 채움(템플릿 또는 AI) |
| **NPC 대화** | 추천 NPC 선택, 회복 대사 받기 | 상태에 맞는 NPC·표정·대사 결정 |
| **세이브** | "오늘을 세이브" | **0완료여도 세이브** + 별사탕 +10 + streak +1 + 도감 체크 |
| **기록** | 주간 리포트·도감 열람 | 누적 집계 |

### 3-4. 특수 분기
- **보스 비움** → 메인이 자기돌봄 퀘스트로 (규칙 정상 동작)
- **7일+ 만에 복귀** → NPC 컴백 대사
- **생존 0완료 3일 연속** → 무기력 개입(비난 아닌 호기심 톤)
- **오프라인** → 체크인·규칙·세이브 로컬 동작, AI 대사만 폴백
- **위기 신호** → 도발 중단, 안전 대사 + 자살예방상담 1393

### 3-5. 구체 예시 — 지수의 어느 오후
> 오후 3시, 아무것도 못 하고 자책 중 → 체크인: HP 버티는 중 / MP 아무 생각 없음 / 집중력 낮음 + {수면 부족, 마감 공포} + 보스 "포트폴리오 정리" → **상태창: 생존 모드, 칭호 '마감 직전 생존자'** → **퀘스트: 메인 "포트폴리오 5분만 들여다보기" / 금지 "오늘 완성하려 하지 않기"** → **힐러: "오늘은 여기까지 온 것만으로 충분해요 🌿"** → 5분 하나 하고 세이브 → "오늘도 저장됨, 별사탕 +10".

## 4. 핵심 설계 결정 (Engineering & Product Decisions)

> 포트폴리오에서 가장 중요한 부분. *무엇을 만들었나*가 아니라 *왜 이렇게 설계했나.*

### 4-1. 규칙 엔진 + LLM 하이브리드
**문제:** NPC 대사·퀘스트를 100% AI로 만들면 비용·일관성·오프라인이 깨지고, 100% 템플릿이면 "내가 입력한 보스"를 못 다룬다.

**결정:** **경계선을 명확히 그었다.**

| 규칙 엔진(결정론) | LLM |
|---|---|
| 난이도 모드 / 퀘스트 슬롯 개수 / 시간박스 / 보상 배율 / 추천 NPC / 추천 루트 | 메인·서브·금지 퀘스트 **문구**, NPC **대사** |

규칙이 "빈 슬롯이 있는 퀘스트 뼈대"를 만들고 LLM은 그 슬롯의 `title`·`flavor`만 채운다. **LLM이 개수·시간·보상을 못 바꾸므로** 일관성·예측가능성·오프라인 동작이 보장된다.
- AI 호출 시점: 상태창 생성 시(퀘스트 일괄, 가벼운 모델) / NPC 대화 시(풍부한 모델)
- 비용 절감: 프롬프트 캐싱 · 구조화 출력(JSON 강제) · 1회 일괄 호출 · 캐시 키 `(status_result_id, npc)`
- 실패 폴백: 규칙 기반 템플릿 + 사전 작성 대사 풀 → **사용자에게 실패를 노출하지 않음**

### 4-2. 상태 → 난이도 규칙 엔진
HP/MP/집중력 각 4단계를 점수화(0~3) → `base = HP+MP+집중력 (0~9)` → `energy = base − 상태이상차감`(하한 0).

| 차감 규칙 | 값 |
|---|---|
| 상태이상 1개당 | −0.5 |
| 중증 태그(수면 부족·현실 도피·마감 공포) 추가 | −0.5 |
| 3개 이상 동시 | 추가 −1.0 |
| 차감 상한 | −4.0 |

| energy 구간 | 모드 | 메인/서브/금지 | 시간박스 | 보상배율 | 위험도 |
|---|---|---|---|---|---|
| `[0.0, 2.1)` | 생존 | 1 / 0 / 2 | 5분 | ×1.5 | 회피 |
| `[2.1, 4.6)` | 쉬움 | 1 / 1 / 1 | 15분 | ×1.2 | 우회 |
| `[4.6, 7.1)` | 보통 | 2 / 2 / 1 | 25분 | ×1.0 | 부분 직면 |
| `[7.1, 9.0]` | 도전 | 3 / 2 / 0 | 45분 | ×0.9 | 정면 |

> **의도된 역설계:** 컨디션이 나쁠수록(생존) 보상 배율을 *올렸다*(×1.5). "조금 한 것"을 크게 인정하기 위해서. 도전은 ×0.9로 인플레 억제.

### 4-3. Zero-failure Streak — 철학을 시스템으로
"세이브=완료 아님"을 **말이 아니라 데이터 제약으로** 보장했다.

| 보장 지점 | 메커니즘 |
|---|---|
| 세이브 가능 조건 | 완료 수와 **무관**. `quests_completed == 0`도 정상 세이브 |
| streak 증가 | 세이브 레코드 존재만 검사(완료 수 미참조) |
| 보상 분리 | 세이브 고정 +10은 완료 보상과 독립 레이어 |
| DB 검증 | `saves` INSERT에 완료 수 체크 제약 **없음**(의도적) |

### 4-4. 데이터 주도 · 제공자 독립 설계
대사·퀘스트·보상·상태이상을 전부 **JSON 데이터 레이어**로 분리했다. 효과:
- **P0는 AI 0회 호출**로 전 루프 동작(데이터만으로)
- LLM 제공자(Grok/Claude/…) **교체 자유** — 구조가 제공자에 의존하지 않음
- 오프라인·AI 실패에도 폴백이 항상 존재

### 4-5. NPC 반응 시스템
상태창(모드·energy·태그·HP)을 입력으로 **추천 NPC → 표정 슬롯 → 대사 카테고리**가 결정론적으로 흐른다. 추천·표정·카테고리·트리거가 **동일한 우선순위 사상을 공유**해 상호 모순이 없게 설계.

## 9. 개발 로드맵

- **P0 — 핵심 루프(AI 없이):** 스플래시·온보딩·체크인·상태창·퀘스트·세이브·홈. 규칙 엔진 + 폴백 데이터만으로 전 루프 동작. Supabase 인증·저장.
- **P1 — AI 레이어:** LLM 연결(퀘스트 문구·NPC 대사) + NPC 상담소·대화. 캐싱·구조화 출력·안전 모더레이션. 푸시 알림.
- **P2 — 리텐션:** 생존 기록·도감·별사탕·streak 마일스톤·NPC 3종 확장·테마/스킨.

## 10. 회고 — 의도적 트레이드오프 (Design Decisions Log)

> 포트폴리오 리뷰어를 위한 "왜 이 길을 택했나" 기록.

- **AI를 일부러 뒤로 미뤘다 (P0는 템플릿).** 데모 속도와 안정성을 위해, AI는 *향상*이지 *전제*가 아니게 설계. AI가 죽어도 앱은 산다.
- **NPC를 5종 다 안 만들고 2종으로 시작.** 감성이 사는지부터 검증. 일러스트·대사·비용이 NPC 수에 비례해 늘기 때문.
- **"완료"를 streak에서 분리.** 가장 큰 제품 베팅. 생산성을 약간 포기하고 *지속*을 택했다. 타깃이 "잘하는 사람"이 아니라 "지친 사람"이라서.
- **금지 퀘스트라는 역발상.** 해야 할 것 목록은 흔하다. 번아웃엔 *하지 말 것*이 더 약이 될 때가 많다는 가설.
- **다크 전용(MVP).** '밤의 여관' 컨셉 정체성 우선. 라이트모드는 v1.2로 의도적 유보.

### 알려진 한계 / 다음 과제
- 폴백 콘텐츠 풀 확장(반복 피로 방지), 대화 선택지·분기 데이터 설계, 2일차 브릿지·컴백 루프 구체화, 에너지 점수 척도 정규화 명세, 도감 집계 컬럼 확정.
- *(상세 감사 결과는 내부 리뷰 로그로 관리 — 의도적으로 "끝난 척"하지 않는다.)*
